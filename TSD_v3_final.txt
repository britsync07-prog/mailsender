Provisions all 200 subdomains with all 4 DNS records via Cloudflare API. No manual configuration ever. Execution Sequence Add domain to Cloudflare — create zone via API (2 minutes per domain) Generate 200 unique DKIM RSA-2048 key pairs — store private keys encrypted in database Run 4 parallel API provisioning threads — SPF, DKIM, DMARC, MX simultaneously Each thread processes at 1,200 API calls per 5 minutes — Cloudflare rate limit respected After all records created — wait 60 minutes for DNS propagation Run verification check — query each record type on all 200 subdomains Set domain.dns_provisioned = true only when ALL records on ALL subdomains verified Alert sent to monitoring dashboard — new domain pool ready Verification Check Query SPF TXT record — verify v=spf1 present and correct IPs listed Query DKIM TXT record — verify public key present and correctly formatted Query DMARC TXT record — verify v=DMARC1 present and rua address correct Query MX record — verify bounce server hostname resolves Any subdomain failing any check — flag for re-provisioning, do not activate 7. Component Technical Specifications 8.1 Email Validation Pipeline In-house built. Every lead passes all 6 stages before entering queue. Runs before warmup. Runs again before every new campaign cycle. # Check Method Pass Result Fail Action 1 Syntax Regex RFC 5322 format check Continue to step 2 Mark invalid — skip all remaining 2 Domain DNS MX record lookup — domain must have mail server Continue Mark invalid — suppress 3 Disposable Check against 50,000+ known disposable domains Continue Mark disposable — suppress 4 Role-based Detect admin@, info@, support@, noreply@, sales@ Continue Mark role_based — suppress 5 Catch-all SMTP probe with random address — 250 = catch-all Mark catch_all — optional use Skip or flag — not suppress 6 SMTP handshake EHLO → MAIL FROM → RCPT TO — check 250 response Mark valid — enter queue Mark invalid — suppress 8.2 Warmup Integration Service Warmbox or Mailreach — minimum 35,000 real accounts in network Why third-party only Internal SMTP-to-SMTP loops are detectable by Gmail ML. Same IP ranges on both ends = synthetic activity flag. Never use internal loops. Connection method API integration — connect all SMTPs to warmup service account Warmup duration Minimum 4 weeks before first cold email sent from any SMTP Warmup runs permanently Service continues running in background after cold email begins — maintains positive engagement signals Week 1-2 volume 2 emails per SMTP per day via warmup service = 400 emails/domain/day Week 3-4 volume 5 emails per SMTP per day via warmup service = 1,000 emails/domain/day Activation criteria SMTP activated for cold email only when warmup_complete = true in database Monitoring during warmup Gmail Postmaster score checked weekly — if below 60 during warmup, extend warmup period 8.3 Content Rotation Engine Every email must appear uniquely human-written. Anti-Fingerprinting Requirements Modern spam filters use ML to detect patterns across sending infrastructure. These requirements prevent any pattern from forming that could link emails across subdomains, domains, or VDS. 9.1 Timing Randomisation All sends distributed across an 8-12 hour window daily — never concentrated at specific times Variable delay between sends per RDP worker — range of 2-30 seconds between jobs No synchronised sending across all 10 VDS — each VDS starts its send window at different offset Send volume not uniform across the day — higher mid-morning and mid-afternoon, lower early morning and late evening Weekday vs weekend sending ratio varies — not identical pattern every day 9.2 Infrastructure Pattern Prevention No two subdomains on same domain send to same recipient within 30 days No two domains on same VDS send to same recipient within 14 days Sending IP rotated across subdomains — no single IP associated with single subdomain permanently RDP worker assignment to subdomains randomised — no fixed worker-to-subdomain mapping User-Agent strings varied across RDP workers — no identical fingerprint 9.3 Content Fingerprint Prevention Zero repeated subject lines across consecutive emails to same industry cluster Zero repeated opening sentences across any two emails in same campaign batch Link domains used in tracking varied — not single tracking domain across all emails HTML structure varied — different div nesting, different class names, different spacing Plain text emails have no consistent formatting pattern — spacing, punctuation varies Email thread history not included — fresh email every time, no forwarded-message artifacts 10.
